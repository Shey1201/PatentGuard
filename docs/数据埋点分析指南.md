# PatentGuard 数据埋点分析指南

## 一、概述

本文档旨在帮助产品经理、开发者和运营人员如何利用 PatentGuard 项目的数据埋点功能，观测用户行为、分析产品使用情况，从而为产品迭代和优化提供数据支撑。

**版本更新 V2.0**：新增管理员首页数据可视化看板支持、扩展埋点事件分析查询模板。

---

## 二、数据埋点能解决什么问题

### 2.1 常见业务问题

| 问题类型 | 具体问题 | 埋点数据如何回答 |
|----------|----------|------------------|
| **用户活跃度** | 用户是否真的在使用系统？| 日活用户数、页面访问量、会话时长 |
| **功能使用率** | 哪些功能被高频使用？| 各功能点击量、使用频率 |
| **用户转化** | 文档上传到审查完成的转化率？| 漏斗分析、步骤流失 |
| **性能问题** | 系统响应是否正常？| API 响应时间、错误率 |
| **用户路径** | 用户在系统中的行为路径？| 用户事件序列分析 |
| **用户分群** | 不同用户群体的行为差异？| 用户分组统计 |

### 2.2 使用场景示例

**场景 1：新功能上线效果评估**

```
问题：新上线的「合同审查」功能使用情况如何？
数据：统计 review_type='contract' 的事件数量，与其他类型对比
```

**场景 2：用户留存分析**

```
问题：注册用户中有多少会再次使用？
数据：对比注册事件和后续 page_view 事件的 user_id
```

**场景 3：性能优化**

```
问题：哪些 API 接口响应慢？
数据：分析 api_logs 表，按 endpoint 分组统计 avg(duration_ms)
```

## 三、数据看板设计

### 3.1 核心指标看板

建议创建以下核心看板：

#### 3.1.1 用户活跃看板

| 指标 | 计算方式 | 数据来源 |
|------|----------|----------|
| DAU（日活用户） | 每日独立用户数 | `SELECT COUNT(DISTINCT user_id) FROM business_events WHERE event_category='navigation'` |
| MAU（月活用户） | 每月独立用户数 | 同上，按月统计 |
| 日会话数 | 每日会话数 | `SELECT COUNT(DISTINCT session_id) FROM business_events` |
| 平均会话时长 | 会话时长均值 | 需计算 page_stay 事件 duration_ms |

#### 3.1.2 页面分析看板

| 指标 | 计算方式 | 数据来源 |
|------|----------|----------|
| PV（页面浏览量） | page_view 事件数 | `COUNT(*) WHERE event_name='page_view'` |
| UV（独立访客） | 独立 session_id 数 | `COUNT(DISTINCT session_id)` |
| 人均浏览页数 | PV / UV | 页面数/访客数 |
| 热门页面 Top10 | 各页面 PV 排序 | 按 page_url 分组统计 |

#### 3.1.3 功能使用看板

| 功能 | 关键事件 | 分析指标 |
|------|----------|----------|
| 文档上传 | document_upload | 上传次数、文件类型分布、文件大小分布 |
| 提交审查 | review_submit | 提交次数、审查类型分布 |
| 审查完成 | review_complete | 完成率、成功率、平均耗时 |
| 知识搜索 | search_action | 搜索次数、无结果率 |

#### 3.1.4 系统性能看板

| 指标 | 计算方式 | 告警阈值建议 |
|------|----------|-------------|
| API 响应时间 P50 | 中位数 | < 500ms |
| API 响应时间 P95 | 95分位数 | < 2s |
| API 错误率 | 错误数/总数 | < 1% |
| 高耗时 API | avg(duration_ms) | > 5s |

## 四、SQL 分析查询模板

### 4.1 用户行为分析

#### 4.1.1 日活用户统计

```sql
-- 日活用户数（过去30天）
SELECT
    DATE(created_at) as date,
    COUNT(DISTINCT user_id) as dau
FROM business_events
WHERE created_at >= NOW() - INTERVAL '30 days'
GROUP BY DATE(created_at)
ORDER BY date;
```

#### 4.1.2 页面访问排行

```sql
-- 最受欢迎的页面 Top 10
SELECT
    properties->>'page_url' as page_url,
    properties->>'page_title' as page_title,
    COUNT(*) as view_count,
    COUNT(DISTINCT session_id) as uv
FROM business_events
WHERE event_name = 'page_view'
GROUP BY page_url, page_title
ORDER BY view_count DESC
LIMIT 10;
```

#### 4.1.3 用户会话分析

```sql
-- 平均会话时长（分钟）
SELECT
    AVG(duration_ms) / 1000 / 60 as avg_session_minutes
FROM business_events
WHERE event_name = 'page_stay';
```

### 4.2 业务转化分析

#### 4.2.1 审查流程漏斗

```sql
-- 审查流程转化漏斗
WITH funnel AS (
    SELECT
        '上传文档' as step,
        COUNT(DISTINCT resource_id) as count
    FROM business_events
    WHERE event_name = 'document_upload'
    AND created_at >= NOW() - INTERVAL '30 days'

    UNION ALL

    SELECT
        '提交审查' as step,
        COUNT(DISTINCT resource_id) as count
    FROM business_events
    WHERE event_name = 'review_submit'
    AND created_at >= NOW() - INTERVAL '30 days'

    UNION ALL

    SELECT
        '审查完成' as step,
        COUNT(DISTINCT resource_id) as count
    FROM business_events
    WHERE event_name = 'review_complete'
    AND created_at >= NOW() - INTERVAL '30 days'
)
SELECT
    step,
    count,
    LAG(count) OVER (ORDER BY step) as prev_count,
    CASE
        WHEN LAG(count) OVER (ORDER BY step) > 0
        THEN ROUND(count::numeric / LAG(count) OVER (ORDER BY step) * 100, 2)
        ELSE 100
    END as conversion_rate
FROM funnel;
```

#### 4.2.2 审查成功率分析

```sql
-- 审查成功率（按类型）
SELECT
    properties->>'review_type' as review_type,
    COUNT(*) as total,
    SUM(CASE WHEN properties->>'success' = 'true' THEN 1 ELSE 0 END) as success_count,
    ROUND(
        SUM(CASE WHEN properties->>'success' = 'true' THEN 1 ELSE 0 END)::numeric
        / COUNT(*) * 100, 2
    ) as success_rate
FROM business_events
WHERE event_name = 'review_complete'
GROUP BY review_type;
```

#### 4.2.3 平均审查耗时

```sql
-- 平均审查耗时（按类型）
SELECT
    properties->>'review_type' as review_type,
    COUNT(*) as count,
    ROUND(AVG((properties->>'duration_ms')::numeric) / 1000, 2) as avg_duration_seconds,
    ROUND(MIN((properties->>'duration_ms')::numeric) / 1000, 2) as min_duration_seconds,
    ROUND(MAX((properties->>'duration_ms')::numeric) / 1000, 2) as max_duration_seconds
FROM business_events
WHERE event_name = 'review_complete'
GROUP BY review_type;
```

### 4.3 用户分群分析

#### 4.3.1 高活跃用户

```sql
-- 过去7天活跃次数Top 10用户
SELECT
    user_id,
    COUNT(*) as event_count,
    COUNT(DISTINCT event_name) as distinct_actions,
    COUNT(DISTINCT DATE(created_at)) as active_days
FROM business_events
WHERE user_id IS NOT NULL
AND created_at >= NOW() - INTERVAL '7 days'
GROUP BY user_id
ORDER BY event_count DESC
LIMIT 10;
```

#### 4.3.2 新用户 vs 老用户

```sql
-- 按用户活跃天数分群
WITH user_activity AS (
    SELECT
        user_id,
        COUNT(DISTINCT DATE(created_at)) as active_days
    FROM business_events
    WHERE user_id IS NOT NULL
    AND created_at >= NOW() - INTERVAL '30 days'
    GROUP BY user_id
)
SELECT
    CASE
        WHEN active_days = 1 THEN '一次性用户'
        WHEN active_days BETWEEN 2 AND 5 THEN '低频用户'
        WHEN active_days BETWEEN 6 AND 15 THEN '中频用户'
        ELSE '高频用户'
    END as user_segment,
    COUNT(*) as user_count,
    ROUND(COUNT(*) * 100.0 / SUM(COUNT(*)) OVER(), 2) as percentage
FROM user_activity
GROUP BY
    CASE
        WHEN active_days = 1 THEN '一次性用户'
        WHEN active_days BETWEEN 2 AND 5 THEN '低频用户'
        WHEN active_days BETWEEN 6 AND 15 THEN '中频用户'
        ELSE '高频用户'
    END
ORDER BY user_count DESC;
```

### 4.4 性能分析

#### 4.4.1 API 响应时间分析

```sql
-- 各 API 端点响应时间统计
SELECT
    endpoint,
    method,
    COUNT(*) as call_count,
    ROUND(AVG(duration_ms)::numeric, 2) as avg_ms,
    ROUND(PERCENTILE_CONT(0.5) WITHIN GROUP (ORDER BY duration_ms)::numeric, 2) as p50_ms,
    ROUND(PERCENTILE_CONT(0.95) WITHIN GROUP (ORDER BY duration_ms)::numeric, 2) as p95_ms,
    ROUND(PERCENTILE_CONT(0.99) WITHIN GROUP (ORDER BY duration_ms)::numeric, 2) as p99_ms
FROM api_logs
WHERE created_at >= NOW() - INTERVAL '7 days'
GROUP BY endpoint, method
ORDER BY avg_ms DESC;
```

#### 4.4.2 API 错误分析

```sql
-- API 错误统计
SELECT
    endpoint,
    status_code,
    COUNT(*) as error_count,
    error_message
FROM api_logs
WHERE status_code >= 400
AND created_at >= NOW() - INTERVAL '7 days'
GROUP BY endpoint, status_code, error_message
ORDER BY error_count DESC;
```

#### 4.4.3 慢请求 Top 20

```sql
-- 最慢的 20 个请求
SELECT
    id,
    method,
    endpoint,
    status_code,
    duration_ms,
    user_id,
    created_at
FROM api_logs
ORDER BY duration_ms DESC
LIMIT 20;
```

## 六、用户行为分析方法论

### 6.1 事件分析

**定义**：统计特定事件的发生次数

**应用场景**：
- 功能点击量统计
- 页面访问量统计
- 业务操作统计

**分析方法**：
```sql
-- 按维度分组统计
SELECT
    event_name,
    properties->>'review_type' as review_type,
    COUNT(*) as count
FROM business_events
GROUP BY event_name, review_type;
```

### 6.2 留存分析

**定义**：分析用户在某个时间后再次访问的比例

**应用场景**：
- 新用户留存
- 功能使用留存
- 周活跃留存

**分析方法**：
```sql
-- 次日留存率
WITH first_day_users AS (
    -- 找出每个用户的首次活跃日期
    SELECT DISTINCT
        user_id,
        DATE(created_at) as first_active_date
    FROM business_events
    WHERE user_id IS NOT NULL
),
retention AS (
    -- 找出每个用户后续的活跃日期
    SELECT
        f.user_id,
        f.first_active_date,
        d.active_date,
        d.active_date - f.first_active_date as days_diff
    FROM first_day_users f
    JOIN (
        SELECT DISTINCT user_id, DATE(created_at) as active_date
        FROM business_events
        WHERE user_id IS NOT NULL
    ) d ON f.user_id = d.user_id
)
SELECT
    days_diff,
    COUNT(DISTINCT user_id) as retained_users
FROM retention
WHERE days_diff > 0
GROUP BY days_diff
ORDER BY days_diff;
```

### 6.3 漏斗分析

**定义**：分析用户在完成目标过程中的各步骤转化情况

**应用场景**：
- 新用户注册流程
- 文档审查流程
- 付费转化流程

**分析方法**：
参见「4.2.1 审查流程漏斗」

### 6.4 路径分析

**定义**：分析用户在系统中的行为路径

**应用场景**：
- 用户导航路径
- 功能使用顺序
- 流失节点识别

**分析方法**：
```sql
-- 用户行为路径序列
WITH session_events AS (
    SELECT
        session_id,
        user_id,
        event_name,
        created_at,
        ROW_NUMBER() OVER (PARTITION BY session_id ORDER BY created_at) as seq
    FROM business_events
    WHERE session_id IN (
        -- 选取特定会话
        SELECT session_id
        FROM business_events
        LIMIT 1000
    )
)
SELECT
    s1.event_name as step1,
    s2.event_name as step2,
    s3.event_name as step3,
    COUNT(*) as path_count
FROM session_events s1
LEFT JOIN session_events s2 ON s1.session_id = s2.session_id AND s2.seq = s1.seq + 1
LEFT JOIN session_events s3 ON s2.session_id = s3.session_id AND s3.seq = s2.seq + 1
GROUP BY s1.event_name, s2.event_name, s3.event_name
ORDER BY path_count DESC
LIMIT 50;
```

### 7.2 常见优化场景

#### 场景 1：发现用户流失严重

**问题**：文档上传后的审查提交率只有 60%

**分析步骤**：
1. 检查 document_upload 事件数量
2. 检查 review_submit 事件数量
3. 对比流失用户的特征（设备、浏览器、地区）
4. 排查是否有技术问题（查看 api_logs 中的错误）

**可能的解决方案**：
- 如果是技术问题 → 修复 Bug
- 如果是体验问题 → 优化上传后的引导
- 如果是功能问题 → 增加提醒通知

#### 场景 2：某页面跳出率高

**问题**：知识库页面跳出率高达 70%

**分析步骤**：
1. 检查 page_view 和 page_stay 数据
2. 分析进入知识库页面的来源（referrer）
3. 检查页面停留时长分布
4. 分析用户下一步行为

**可能的解决方案**：
- 优化页面内容质量
- 增加内部链接引导
- 优化页面加载速度

#### 场景 3：API 响应慢

**问题**：审查 API 的 P95 响应时间超过 10 秒

**分析步骤**：
1. 检查慢请求的分布时段
2. 分析是否与特定文档大小相关
3. 检查数据库查询性能
4. 分析 LLM 调用耗时

**可能的解决方案**：
- 增加超时处理
- 优化后端算法
- 增加缓存机制
- 考虑异步处理

### 7.3 优化效果验证

**A/B 测试流程**：
1. 设定优化目标和指标
2. 设计实验方案
3. 分配流量
4. 收集数据
5. 统计分析结果

```sql
-- 对比实验组和对照组的转化率
SELECT
    properties->>'ab_group' as group_name,
    COUNT(DISTINCT resource_id) as total_users,
    SUM(CASE WHEN properties->>'converted' = 'true' THEN 1 ELSE 0 END) as converted,
    ROUND(
        SUM(CASE WHEN properties->>'converted' = 'true' THEN 1 ELSE 0 END)::numeric
        / COUNT(DISTINCT resource_id) * 100, 2
    ) as conversion_rate
FROM business_events
WHERE event_name = 'experiment'
GROUP BY group_name;
```

## 九、常见问题排查

### 9.1 数据不准确

**问题**：统计数据与预期差异大

**排查步骤**：
1. 检查埋点代码是否正确执行
2. 验证事件名称和属性是否一致
3. 检查时间范围是否正确
4. 确认是否存在数据重复

### 9.2 数据延迟

**问题**：新事件迟迟未出现在统计中

**排查步骤**：
1. 检查前端 SDK 是否正常工作
2. 检查网络请求是否成功
3. 验证后端 API 是否正常接收
4. 检查数据库写入是否有延迟

### 9.3 数据丢失

**问题**：部分事件未记录

**排查步骤**：
1. 检查前端控制台是否有错误
2. 验证事件队列是否已满导致丢弃
3. 检查网络断开时的重试机制

#### 10.2.3 结果查看转化率

```sql
-- 从审查完成到查看结果的转化率
WITH review_complete AS (
    SELECT
        resource_id,
        MIN(created_at) as complete_at
    FROM business_events
    WHERE event_name = 'review_complete'
    AND created_at >= NOW() - INTERVAL '30 days'
    GROUP BY resource_id
),
review_view AS (
    SELECT
        resource_id,
        MIN(created_at) as view_at
    FROM business_events
    WHERE event_name = 'review_view_result'
    AND created_at >= NOW() - INTERVAL '30 days'
    GROUP BY resource_id
)
SELECT
    COUNT(DISTINCT rc.resource_id) as total_completed,
    COUNT(DISTINCT rv.resource_id) as total_viewed,
    ROUND(
        COUNT(DISTINCT rv.resource_id)::numeric
        / NULLIF(COUNT(DISTINCT rc.resource_id), 0) * 100, 2
    ) as view_rate
FROM review_complete rc
LEFT JOIN review_view rv ON rc.resource_id = rv.resource_id
AND rv.view_at > rc.complete_at;
```
