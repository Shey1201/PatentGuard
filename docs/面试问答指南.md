## 一、产品/业务方向

### 1.1 基础认知类

---

#### Q1：请用一句话介绍 PatentGuard 是什么产品？

**参考答案：**

> PatentGuard 是一款基于 RAG（检索增强生成）技术的企业级文档合规审查 SaaS 系统，聚焦专利申请书、法律法规、合同协议、政策文件四类文档，实现「上传即审查、结果可溯源」的智能合规审查。

**回答思路：**

- 先说产品形态（Web SaaS）
- 再说核心技术（RAG + LLM）
- 最后说用户价值（效率提升 + 结果可溯源）
- 如果时间充裕，可以补充目标用户（企业法务/专利工程师）

---

#### Q2：PatentGuard 解决的是什么痛点？

**参考答案：**

| 痛点 | 我们怎么解决 |
|------|-------------|
| 人工审查效率低 | AI 自动化，分钟级出结果，相比人工效率提升 90%+ |
| 审查标准不统一 | 知识库统一规则输出，审查结论一致性有保障 |
| 缺乏可追溯依据 | 每条风险点标注法规来源，结果可解释可验证 |
| API 配置不灵活 | 支持用户自带 API Key，多 LLM 提供商可选 |

**回答思路：**

- 列举 3-4 个核心痛点
- 每个痛点对应一个解决方案
- 强调「效率 + 标准 + 可信」三个关键词

---

**参考答案：**

| 合规部门 | 法规政策合规性审查 |

#### Q4：项目的商业模式是什么？

**参考答案：**

> 我们的商业模式是 SaaS 订阅 + 按用量计费。基础版按月/年订阅，高级版按审查文档数量计费。此外支持企业私有化部署，按项目制收费。

**回答思路：**

- SaaS 订阅是主流
- 可以加「用户自带 API Key」的灵活商业模式
- 如果面试官追问盈利，可以结合成本（LLM API 调用费 + 服务器）说

#### Q11：通用 AI（ChatGPT/Claude）能替代你们的产品吗？为什么？

**参考答案：**

**通用 AI 的三大问题：**

| 问题 | 说明 |

| **知识有截止日期** | 无法知道最新法规 |
| **幻觉问题严重** | 可能编造法规条文 |

**RAG vs 通用 AI：**

| 维度 | 通用 AI | RAG 系统 |
|------|---------|----------|
| 知识来源 | 训练数据 | 企业知识库 |
| 知识时效性 | 有截止日期 | 实时更新 |
| 引用溯源 | ❌ 无法 | ✅ 可以 |
| 准确性 | 一般 | 高（可校验）|

> 类比：**通用 AI 像全科医生，我们像专科医生**——企业级场景需要「专科医生」。

## 1.3 产品设计类（8 问题）

---

#### Q15：为什么选择做文档合规审查这个方向？

**参考答案：**

> 选择这个方向有三个原因：

**一、市场需求层面：**

> - 中国企业法务人员约 50 万，人均每天处理文档超 10 份
> - 一份合同的人工审查成本约 ¥200-500，付费意愿强
> - 80% 的法务部门表示「审查效率低」是最大痛点

**二、技术可行性层面：**

> - GPT-4、Claude 等模型在文本理解、结构化输出方面已经非常成熟
> - RAG 技术解决了知识时效性和幻觉问题
> - Embedding 模型成本大幅下降

**三、差异化可能性层面：**

> 通用 AI 在法务场景有三个问题：
>
> 1. 知识有截止日期，不知道最新法规
> 2. 无法给出具体的法规引用
> 3. 幻觉问题严重
>
> 这些问题是「通用」解决不了的，必须靠「垂直」解决。

---

#### Q16：为什么用 RAG 而不是直接让 LLM 审查？

**参考答案：**

**直接用 LLM 的三大问题：**

| 问题 | 说明 |
|------|------|
| **知识时效性** | LLM 的知识有截止日期 |
| **幻觉问题** | LLM 会「一本正经地胡说八道」 |
| **缺乏引用溯源** | 无法给出具体法规引用 |

**RAG 的核心优势：**

| 优势 | 说明 |
|------|------|
| **知识实时** | 知识库可以实时更新 |
| **引用溯源** | 每条结论都标注来源 |
| **可验证** | 用户可以核对引用 |
| **可控制** | 知识库内容可控 |

**我们的创新：引用校验**

> 普通 RAG 系统有一个问题：**LLM 可能捏造引用**。
>
> 我们的解决方案：**二次校验**——LLM 输出结论后，再用关键词匹配验证引用是否真实存在于知识库中。如果不存在，就标记为「⚠️ 引用未经知识库验证」。

---

#### Q17：产品的 MVP 是什么？为什么这样设计 MVP？

**参考答案：**

**MVP 功能定义：**

| 模块 | MVP 功能 | 设计理由 |
|------|----------|----------|
| 用户认证 | 注册/登录/JWT Token | 产品基础能力 |
| 知识库 | 文档上传、解析、向量化、分类 | RAG 基础能力 |
| 文档审查 | RAG + LLM 审查、结果展示 | 核心价值所在 |
| 系统配置 | LLM API 配置 | 灵活切换模型 |

**MVP 设计原则：**

1. **核心价值优先**——只做「文档审查」一件事，做到极致
2. **技术风险可控**——RAG 全链路跑通后，再逐步加功能
3. **快速验证**——2-4 周出 MVP，快速投放验证

**MVP 验证指标：**

| 指标 | 目标值 |
|------|--------|
| 审查准确率 | ≥ 80% |
| 用户满意度 | ≥ 4/5 |
| 审查耗时 | ≤ 10 分钟 |
| 引用准确率 | ≥ 90% |

#### Q18：产品有哪些审查类型？为什么这样设计？

**参考答案：**

**四种审查类型：**

| 审查类型 | 适用场景 | 审查重点 |
|----------|----------|----------|
| **通用审查** | 各类文档全面检查 | 综合合规性 |
| **专利审查** | 专利申请书、权利要求书 | 新颖性、创造性、实用性 |
| **法律审查** | 法律法规、政策文件 | 合规性、时效性 |
| **合同审查** | 合同协议、条款文本 | 风险点、缺失条款 |

**设计依据：**

> 调研了 10 家企业的法务部门：
>
> - 40% 是「合同审查」
> - 30% 是「专利审查」
> - 20% 是「法规审查」
> - 10% 是「其他」

> 不同类型文档的关注点不同，Prompt 需要差异化。

---

#### Q19：知识库在产品中扮演什么角色？为什么知识库是核心壁垒？

**参考答案：**

**知识库的产品角色：**

| 角色 | 说明 | 产品价值 |
|------|------|----------|
| **检索基础** | 审查时从知识库检索相关法规 | 保证检索有内容可查 |
| **上下文来源** | 为 LLM 提供准确的法律依据 | 保证生成有据可依 |
| **一致性保障** | 所有审查基于同一知识库 | 保证结论标准统一 |
| **可溯源依据** | 每条风险点都来自知识库 | 保证结果可验证 |

> 类比：**LLM 是大脑，知识库是图书馆**

**知识库作为核心壁垒的原因：**

| 壁垒类型 | 壁垒强度 |
|----------|----------|
| **数据积累** | 高质量法规知识库需要 2-3 年积累 |
| **时效性** | 法规不断更新，需要持续维护 |
| **准确性** | 法规文本需要专业人士校验 |
| **覆盖面** | 需要覆盖多种法规类型 |

#### Q6：为什么用 RAG 而不是直接让 LLM 审查？

**参考答案：**

> 直接让 LLM 审查有两个问题：一是 LLM 知识有截止日期，可能不知道最新法规；二是容易产生幻觉，捏造不存在的法规依据。
>
> RAG 的优势是：**答案来源于你的知识库**，知识库里的法规是实时的、准确的。而且 RAG 支持引用溯源，用户可以看到每条结论的来源。
>
> 我们的系统还有一个关键设计：**引用校验**。如果 LLM 输出的法规引用在知识库中找不到对应内容，会标记为「未经验证」，防止误导用户。

**回答思路：**

- 先说问题（知识时效性 + LLM 幻觉）
- 再说 RAG 的解决方案
- 重点强调「引用校验」是差异化亮点

#### Q19：什么是 RAG？PatentGuard 的 RAG 链路是怎样的？

**参考答案：**

> RAG（Retrieval-Augmented Generation，检索增强生成）是一种结合检索和生成的 AI 架构。核心思想是：**让 LLM 基于检索到的准确信息来回答问题**，而不是仅依赖 LLM 的内部知识。
>
> **PatentGuard 的 RAG 链路：**
>
> ```
> 知识库构建阶段：

>
> 文档审查阶段：
> 用户上传待审文档 → 取前4000字符作为查询 → 生成查询向量
>     → pgvector 检索(top_k=5) → 构建上下文
>     → LLM 审查 → 引用校验 → 返回结果
> ```

- 先解释 RAG 是什么

#### Q20：文档为什么要分块（Chunking）？分块大小怎么定的？

**参考答案：**

> 文档分块的原因：
>
> 1. **控制 Embedding 精度**：一段文字太长，语义会稀释；太短则缺乏上下文。
> 2. **避免 LLM Context 溢出**：LLM 有 token 上限，分块后可以灵活组合。
> 3. **提高检索精度**：小块更容易和查询匹配，召回率更高。
>
> **分块大小怎么定：**
>
> 我们用的是 **512 字符 + 50 重叠**。原因：
>
> - 512 字符约等于 128-170 个 tokens，适合大多数 Embedding 模型
> - 50 字符重叠保证块之间的语义连贯
> - 实际测试中，这个配置召回率最高

**回答思路：**

- 先说为什么分块
- 再解释具体参数
- 可以说「实际测试调优」体现工程实践

#### Q21：向量检索用的什么算法？为什么要用 HNSW？

**参考答案：**

> 我们用的是 **HNSW（Hierarchical Navigable Small World，分层可导航小世界）** 索引，基于余弦相似度计算。
>
> **为什么选 HNSW：**
>
> | 对比维度 | HNSW | IVF（倒排索引） |
> |----------|------|----------------|
> | 查询速度 | O(log n)，极快 | O(log n + k)，较快 |
> | 构建速度 | 较慢 | 快 |
> | 内存占用 | 高 | 中 |
> | 适用规模 | 百万级 | 亿级 |
>
> 我们的数据规模（万级文档），HNSW 完全够用。而且 pgvector 的 HNSW 实现很成熟，配置简单。

**回答思路：**

- 先说用的什么

- 结合自己项目的规模说明选型理由

#### Q22：向量检索用的什么距离度量？为什么？

**参考答案：**

> 我们用的是 **余弦相似度（Cosine Similarity）**。
>
> 公式：`similarity = 1 - cosine_distance`
>
> **为什么选余弦相似度：**
>
> 1. **适合文本语义**：余弦相似度衡量的是「方向」，适合比较两个文本的语义是否相似
> 2. **向量长度不敏感**：不管文本长短，只要语义相近，相似度就高
> 3. **pgvector 原生支持**：`embedding <=> vector` 就是计算余弦距离

- 解释为什么适合文本检索

#### Q24：什么是引用校验？为什么需要它？

#### Q29：为什么用 pgvector 而不是专门的向量数据库（如 Pinecone）？

**参考答案：**

| 对比维度 | pgvector | Pinecone |
|----------|----------|----------|
| 运维成本 | 低（PostgreSQL 一体）| 高（独立服务）|
| 数据规模 | 万~百万级 | 百万~亿级 |
| 功能完整性 | SQL + 向量一体 | 纯向量检索 |
| 成本 | 低（云数据库费用）| 按容量计费 |
| 适用场景 | 中小规模、需关联查询 | 超大规模、纯向量场景 |

> 我们的数据规模（万级文档）pgvector 完全够用。选择 pgvector 的原因：
>
> 1. **运维简单**：不需要维护额外服务，直接用 PostgreSQL
> 2. **功能一体**：文档元数据、分块、向量可以用 SQL JOIN 查询
> 3. **成本低**：不需要额外的向量数据库服务费用
> 4. **HNSW 支持**：pgvector 已经实现了高效的 HNSW 索引

#### Q38：什么是 Prompt Engineering？为什么重要？有哪些最佳实践？

**参考答案：**

**Prompt Engineering 的定义：**

> Prompt Engineering（提示工程）是指**设计和优化输入提示词，以获得更好输出的实践**。
>
> 在 RAG 系统中，Prompt 是用户和 LLM 之间的「翻译官」——把用户意图翻译成 LLM 能理解的形式。

**为什么重要：**

| 原因 | 说明 |
|------|------|
| **模型不听使唤** | 同样的意思，不同的 Prompt 效果差很多 |
| **成本敏感** | 好的 Prompt 减少 token 消耗 |
| **延迟敏感** | 简洁的 Prompt 响应更快 |
| **准确性关键** | Prompt 直接影响输出准确性 |

**最佳实践：**

**一、Prompt 结构设计：**

```
┌─────────────────────────────────────────────────────────────┐
│                    Prompt 最佳结构                           │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│ 【角色定义】你是一个专业的专利合规审查专家                   │
│                                                             │
│ 【任务说明】请审查以下文档，判断是否符合专利法规定            │
│                                                             │
│ 【上下文】相关法规：{context}                                │
│                                                             │
│ 【输入】待审文档：{document}                                │
│                                                             │
│ 【输出格式】JSON 格式，包含 compliance、risk_level 等字段    │
│                                                             │
│ 【约束】只基于提供的上下文回答，不要编造                      │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

**二、关键技巧：**

#### Q39：什么是 Fine-tuning（微调）？什么时候用 Fine-tuning 而不是 Prompt Engineering？

**参考答案：**

**Fine-tuning vs Prompt Engineering：**

| 维度 | Prompt Engineering | Fine-tuning |
|------|-------------------|--------------|
| **定义** | 优化输入提示词 | 在预训练模型上继续训练 |
| **成本** | 低（只改文字）| 高（需要 GPU 训练）|
| **时间** | 快（分钟级）| 慢（小时/天级）|
| **数据需求** | 少量示例 | 大量标注数据 |
| **适用场景** | 通用任务、快速迭代 | 垂直场景、长期优化 |
| **灵活性** | 高（随时调整）| 低（需要重新训练）|

**什么时候用 Fine-tuning：**

| 场景 | 说明 | 示例 |
|------|------|------|
| **垂直领域专精** | 需要深度理解特定领域 | 医疗、法律、金融 |
| **风格固定** | 需要统一的输出风格 | 特定格式、术语 |
| ** Prompt 不够用** | Prompt 优化到极限效果仍不好 | 复杂推理、专业判断 |
| **成本优化** | 大量调用，大模型太贵 | 用小模型替代 |

**什么时候用 Prompt Engineering：**

| 场景 | 说明 | 示例 |
|------|------|------|
| **通用任务** | 不需要深度专业知识 | 简单问答、翻译 |
| **快速验证** | 需要快速验证假设 | MVP 阶段 |
| **灵活多变** | 任务经常变化 | 多场景适配 |
| **资源有限** | 没有 GPU、数据 | 初创公司 |

**我们的选择：**

> 在 PatentGuard 项目中，我们选择了 **Prompt Engineering**，原因：
>
> 1. **快速验证**：Prompt 调整是分钟级，可以快速迭代
> 2. **成本低**：不需要训练成本
> 3. **够用**：基于 GPT-4/Claude 等大模型，Prompt Engineering 效果已经很好
> 4. **灵活**：可以随时调整 Prompt 适配不同场景

**未来规划：**

> 未来如果需要微调，可能的方向：
>
> - 训练一个「合规审查专用模型」
> - 用大量审查数据微调小模型（如 Llama）
> - 降低推理成本

- 再讲我们为什么选择 Prompt Engineering

#### Q41：你怎么理解 AI 产品的用户信任问题？怎么建立用户信任？

**参考答案：**

**用户信任的重要性：**

> AI 产品的最大挑战不是「能不能做到」，而是「用户敢不敢用」。
>
> 在法务、医疗、金融等高风险场景，错误的后果很严重。用户需要**可信赖的 AI**，而不是「最好用的 AI」。

**信任缺失的原因：**

| 原因 | 说明 | 示例 |
|------|------|------|
| **幻觉问题** | AI 可能生成错误信息 | 捏造法规条文 |
| **黑盒问题** | 用户不知道 AI 怎么得出结论 | 「机器说了算」|
| **责任不清** | AI 错了谁负责 | 「出了事找谁」|
| **能力边界不明** | 用户不知道 AI 能做什么不能做什么 | 过度依赖 |

**建立用户信任的策略：**

**一、透明可解释**

| 策略 | 说明 | 产品体现 |
|------|------|----------|
| **引用溯源** | 展示结论的依据来源 | 点击查看引用原文 |
| **置信度展示** | 展示结论的可靠程度 | 高/中/低置信度 |
| **不确定性标记** | 标记不确定的内容 | ⚠️ 未经验证 |
| **过程可见** | 展示 AI 分析过程 | 检索到哪些文档 |

**二、可控可干预**

| 策略 | 说明 | 产品体现 |
|------|------|----------|
| **用户反馈** | 允许用户纠正错误 | 👍👎 反馈按钮 |
| **人工复核** | 支持人工复核流程 | 导出报告人工审核 |
| **参数调整** | 允许用户调整参数 | 审查类型选择 |

**三、责任明确**

| 策略 | 说明 | 产品体现 |
|------|------|----------|
| **免责声明** | 明确 AI 是辅助不是替代 | 使用前告知 |
| **使用指南** | 告知正确的使用方式 | 新手引导 |
| **适用范围** | 说明适合的使用场景 | 场景说明 |

**四、持续改进**

| 策略 | 说明 | 产品体现 |
|------|------|----------|
| **用户反馈闭环** | 用户反馈能被处理 | 工单系统 |
| **持续优化** | 基于反馈持续改进 | 版本更新说明 |
| **公开透明** | 公开更新日志和改进 | 更新公告 |

**我们的实践案例：**

> **引用校验机制**——这是我们建立用户信任的核心功能。
>
> 1. **展示引用**：每条结论都标注来源法规
> 2. **可点击**：用户可以查看引用原文
> 3. **可验证**：不确定的引用标记为「⚠️」
> 4. **可反馈**：用户可以标记错误的引用
>
> 效果：上线引用校验后，NPS 提升 15 分，用户开始主动推荐。

> - AI 提供分析和建议

#### Q46：怎么理解 AI 产品的数据飞轮？数据的重要性是什么？

**参考答案：**

**什么是数据飞轮？**

> **数据飞轮**：产品产生数据，数据优化模型，模型提升产品，产品吸引更多用户，更多用户产生更多数据，形成正向循环。
>
> ```
>        ┌─────────────┐
>        │   用户增长   │
>        └──────┬──────┘
>               ↓
>        ┌─────────────┐
>        │  产生数据   │
>        └──────┬──────┘
>               ↓
>        ┌─────────────┐
>        │  优化模型   │
>        └──────┬──────┘
>               ↓
>        ┌─────────────┐
>        │  提升产品   │
>        └──────┬──────┘
>               ↓
>        ┌─────────────┐
>        │  吸引用户   │ ←──────┐
>        └─────────────┘        │
>               ↑               │
>               └───────────────┘
> ```

**为什么数据飞轮重要？**

| 价值 | 说明 |
|------|------|
| **竞争优势** | 数据越多，模型越好，优势越大 |
| **先发优势** | 早期用户数据是壁垒 |
| **马太效应** | 强者更强，弱者更弱 |
| **护城河** | 数据壁垒难以复制 |

**我们的数据飞轮实践：**

> **数据收集：**
>
> - 用户审查行为数据（查询词、点击、反馈）
> - 审查结果数据（准确率、满意度）
> - 知识库使用数据（哪些法规被检索最多）
>
> **数据应用：**
>
> - 优化检索算法（提升召回率）
> - 优化 Prompt（提升准确率）
> - 优化知识库（补充高频需求）
>
> **数据壁垒：**
>
> - 审查历史数据是壁垒
> - 知识库更新是壁垒
> - 用户习惯是壁垒

**数据收集的挑战：**

| 挑战 | 说明 | 应对策略 |
|------|------|----------|
| **隐私问题** | 用户数据不能乱用 | 数据脱敏、授权同意 |
| **标注成本** | 高质量数据需要标注 | 众包、用户反馈 |
| **数据质量** | 垃圾数据没价值 | 数据清洗、质量把控 |
| **数据孤岛** | 数据分散难以整合 | 数据平台建设 |

**数据飞轮 vs 模型迭代：**

| 维度 | 数据飞轮 | 模型迭代 |
|------|----------|----------|
| 核心 | 数据积累 | 技术突破 |
| 速度 | 渐进式 | 跳跃式 |
| 壁垒 | 长期积累 | 可能被颠覆 |
| 成本 | 运营成本 | 研发成本 |

**回答思路：**

- 先解释什么是数据飞轮
- 再讲为什么重要
- 再讲我们的实践
- 再讲挑战和应对
- 体现你对「数据战略」的理解

#### Q47：怎么理解 AI 产品的护城河？和技术壁垒有什么区别？

**护城河 vs 技术壁垒：**

| 维度 | 技术壁垒 | 护城河 |

| **定义** | 技术上的领先优势 | 综合竞争壁垒 |
| **持久性** | 技术可能被追平 | 更持久 |
| **可复制性** | 技术可以被复制 | 难以复制 |
| **范围** | 狭窄 | 宽广 |
| **举例** | RAG 链路 | 品牌、数据、生态 |

**AI 产品的护城河类型：**

**一、数据护城河**

| 类型 | 说明 | 壁垒强度 |
|------|------|----------|
| **独家数据源** | 别人没有的数据 | ⭐⭐⭐⭐⭐ 极强 |
| **数据规模** | 大量用户行为数据 | ⭐⭐⭐⭐ 强 |
| **数据质量** | 高质量标注数据 | ⭐⭐⭐⭐ 强 |

| 类型 | 说明 | 壁垒强度 |

| **数据网络** | 数据越多模型越好 | ⭐⭐⭐⭐ 强 |
| **生态网络** | 合作伙伴越多越有价值 | ⭐⭐⭐⭐ 强 |

**三、转换成本护城河**

| 类型 | 说明 | 壁垒强度 |
|------|------|----------|
| **习惯成本** | 用户习惯了你的产品 | ⭐⭐⭐ 中 |
| **数据迁移** | 数据迁移成本高 | ⭐⭐⭐⭐ 强 |
| **培训成本** | 重新培训用户成本高 | ⭐⭐⭐ 中 |

**四、品牌护城河**

| 类型 | 说明 | 壁垒强度 |
|------|------|----------|
| **认知度** | 品牌知名度高 | ⭐⭐⭐ 中 |
| **信任度** | 用户信任品牌 | ⭐⭐⭐⭐ 强 |
| **口碑** | 好的口碑传播 | ⭐⭐⭐⭐ 强 |

**我们产品的护城河分析：**

| 护城河类型 | 我们的情况 | 建立难度 |

| **技术护城河** | RAG + 引用校验 | ⭐⭐⭐ 中 1-2 年 |
| **网络护城河** | 早期建设中 | ⭐⭐⭐⭐ 持续 |
| **品牌护城河** | 早期建设中 | ⭐⭐⭐⭐⭐ 长期 |

**如何构建护城河：**

> **护城河构建策略：**
>
> 1. **早期**：建立技术壁垒（RAG 链路、引用校验）
> 2. **中期**：积累数据壁垒（知识库、用户数据）
> 3. **后期**：建立品牌壁垒（口碑、行业影响力）
> 4. **最终**：形成网络效应（用户越多越有价值）

### 4.2 产品定位话术

```
一句话定义：PatentGuard 是一款基于 RAG 技术的文档合规审查 SaaS 系统。

核心价值：我们解决了三个核心问题：
1. 效率问题——人工审查一份合同需要数小时，我们实现分钟级自动审查
2. 标准问题——通过知识库统一规则，输出结论一致
3. 可信问题——每条风险点都标注法规来源，结果可溯源

目标用户：企业法务、专利工程师、合规部门、律所
```

### 4.3 技术架构话术

```
整个系统分为三层：

前端层：React 18 + TypeScript + Vite + Ant Design
      - Vite 做开发代理，解决跨域问题
      - Zustand 做状态管理，Ant Design 做 UI 组件

后端层：FastAPI + SQLAlchemy + pgvector
      - FastAPI 处理 API 请求，异步高性能
      - SQLAlchemy 做数据库 ORM
      - pgvector 做向量检索

数据层：PostgreSQL + pgvector + OpenAI API
      - PostgreSQL 存元数据
      - pgvector 存向量，支持 HNSW 索引
      - OpenAI 做 LLM 调用

RAG 链路：用户上传文档 → 解析 → 分块 → 向量化 → 检索 → LLM 审查 → 引用校验 → 结果
```

---

### 4.4 项目亮点话术

```
我觉得这个项目有三个亮点：

第一，RAG 链路完整。从文档解析、分块、向量化、检索到 LLM 分析，全链路跑通。

第二，引用校验机制。很多 RAG 系统直接让 LLM 给结论，容易捏造法规依据。我们实现了二次校验，如果引用不在知识库中，就标记为不可信。这在法务场景非常重要。

第三，多 LLMs 支持。用户可以自带 API Key，灵活选择 OpenAI、Claude、通义等模型，按用量计费。
```

---

### 4.5 遇到挑战话术

```
项目最大的挑战是 LLM 输出的稳定性问题。

最初我们发现，LLM 有时会编造法规条文，比如引用「专利法第 XX 条」，但知识库中根本没有这条。

我们分析原因：LLM 的知识是通用的，不一定精准匹配法务领域的专业表述。

解决方案：实现了引用校验机制。每次 LLM 输出法规引用后，用关键词匹配去知识库验证，找不到就标记为「未经验证」。同时优化了 Prompt，要求「只基于知识库内容输出结论」。

效果：引用捏造问题大幅减少，用户信任度显著提升。
```
